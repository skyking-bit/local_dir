Memory Types and Characteristics
Veloce Strato supports several physical memory types, each with distinct capacity, port, and muxing characteristics.
Memory Type Instances / Unit Size Read Ports Write Ports Muxing Notes
XMEM 560 / crystal 1 KB (1K × 8 bits) 1 1 No No muxing possible
CMEM 64 / crystal 256 KB (64K × 32 bits) Up to 16R Up to 16W Yes Up to 16R/16W with muxing
LCMEM 4 / crystal 4 MB (1M × 32 bits) Up to 16R Up to 16W Yes Up to 16R/16W with muxing
SBMEM 4 / SXB (32 SXBs/StratM) 1 GB (64M × 128 bits) 1 1 No No mixing possible
⚡ XMEM — Small & Fast
560 instances per crystal. Ideal for small, high-speed memories. 1KB per instance with single read/write ports. No muxing capability.
🔀 CMEM — Muxable Medium
64 instances per crystal with 256KB each. Supports up to 16 read and 16 write ports via muxing — great for multi-port memory needs.
📦 LCMEM — Large Capacity
4 instances per crystal at 4MB each. Same muxing flexibility as CMEM but with much larger capacity for system-level memories.
🏢 SBMEM — Massive Storage
4 instances per SXB with 1GB each. Used for very large system memories. No mixing possible. Supports comodel transaction-based upload/download.

Using Parameterized Memory Interface
Veloce recognizes a module as a memory when it has the VMW_PARAM_MEM prefix and exactly six parameters in the correct order.
ℹ️ When any non-zero number of parameters are passed to a module without the VMW_PARAM_MEM prefix, velsyn reports a warning. When any number of parameters other than six is passed to a module with the prefix, velsyn reports an error.

Six Required Parameters (in order)

#ParameterDescription
1address_widthWidth of address bus
2data_widthWidth of data bus
3write_semanticse.g. VMW_MEM_RISING_EDGE, VMW_MEM_FALLING_EDGE
4output_enableVMW_MEM_NO_OUTPUT_ENABLE or VMW_MEM_OUTPUT_ENABLE
5num_read_portsNumber of read ports
6num_write_portsNumber of write ports

Memory Module Terminal Names

Terminals use the following naming convention where n is the read or write port number:

DirectionTerminalWidthDescription
input RAn [addr_width-1:0] Read Address port n
input WAn [addr_width-1:0] Write Address port n
output RDn [data_width-1:0] Read Data port n
input WDn [data_width-1:0] Write Data port n
input WEn 1 bit Write Enable port n
input OEn 1 bit Output Enable port n
input CWEn 1 bit Conditional Write Enable
input CLKn 1 bit Clock port n
input BEn [b-1:0] Byte Enable (b = data_width in bytes)
Two-Read-Port, One-Write-Port Memory Instantiation
VERILOG
// Infers Two-Read-Port, One-Write-Port Memory VMW_PARAM_MEM_mem #(8, 16, `VMW_MEM_RISING_EDGE, `VMW_MEM_NO_OUTPUT_ENABLE, 2, 1) mem_instance ( .RA0(ra0), .RD0(rd0), .RA1(ra1), .RD1(rd1), .WA0(wa), .WD0(wd), .WE0(we) );
💡 The generated module interface will have: RAO, RDO, RA1, RD1, WAO, WDO, WEO — read/write ports numbered independently starting at port zero.
VHDL Parameterized Memory Instantiation
VHDL
-- First instantiation: AddrW=16, DataW=8, 2 write ports, 1 read port g1: VMW_PARAM_MEM_MY_MEM generic map (AddrW => 16, DataW => 8) PORT MAP ( RA0 => address1, WA0 => address1, WA1 => address2, WE0 => we1, WE1 => we2, RD0 => out_data1, WD0 => data1, WD1 => data2 ); -- Second instantiation: same component, AddrW=8 g2: VMW_PARAM_MEM_MY_MEM generic map (AddrW => 8, DataW => 8) PORT MAP ( RA0 => address1(7 downto 0), WA0 => address2(7 downto 0), WA1 => address2(15 downto 8), WE0 => we1, WE1 => we2, RD0 => out_data2, WD0 => data1, WD1 => data2 );
💡 VHDL parameterized memory does not require entity/architecture definitions. Only address and data widths are parameterizable. RTLC automatically treats these as memory candidates.
Bit-Writable RTL Parameterized Memory Wrapper
VERILOG
module RAM_BITWR_12_32_FALLING (rd_a, rd_d, wr_a, wr_d, enable); input [11:0] rd_a; input [11:0] wr_a; input enable; output [31:0] rd_d; input [31:0] wr_d; write_mem #(12, 32, `VMW_MEM_FALLING_EDGE, 1) mem ( .ra(rd_a), .rd(rd_d), .wa(wr_a), .wd(wr_d), .we(enable) ); endmodule
✏️
Write Semantics
Write semantics define when a memory write takes effect relative to the clock edge or level signal.
⬆️ VMW_MEM_RISING_EDGE
Write is triggered on the rising edge of the clock. Most common write semantic for synchronous memories.
Synchronous Most Common
⬇️ VMW_MEM_FALLING_EDGE
Write is triggered on the falling edge of the clock. Used in designs that latch data on the negative clock edge.
Synchronous Falling Edge
〰️ Level-Sensitive
Write occurs while the write enable signal is asserted high. Used for latch-based or asynchronous memory designs.
Asynchronous Level-Ordered

Output Enable Types

ConstantRead Data Terminal TypeUse Case
VMW_MEM_NO_OUTPUT_ENABLE Declared as output Standard read port — always drives output
VMW_MEM_OUTPUT_ENABLE Declared as inout Tri-state read port — output enable controls driving
⚠️ If you use a memory module name twice, this implies they have the same semantics. Only the width and depth can differ. Module names must be consistent across the design.

Memory Content Initialization
Memory content initialization for RTL, VHDL, and Gates. Initialization comes from the rtl.out/vmw_mem_contents file.
📂 $readmemb / $readmemh in Initial Block
▼

RTLC supports Verilog $readmemb and $readmemh tasks for initialization of memories within a local scope. All instances of the memory receive the same file by default.

Local scope $readmemh
VERILOG
module Mod (....); initial $readmemh("init.txt", mem); endmodule module Top (....); Mod m1 (); // → reads init.txt Mod m2 (); // → reads init.txt endmodule
🔗 Hierarchical Memory Initialization
▼

Hierarchical initialization takes precedence over generic initialization. A specific instance can be given a different file than the module default.

Hierarchical override
VERILOG
module Top (....); initial $readmemh("init1.txt", Top.m1.mem); // hierarchical — overrides Mod m1 (); // → reads init1.txt (hierarchical takes precedence) Mod m2 (); // → reads init.txt (module default) endmodule
🔁 Explicit Assignment / Loop in Initial Block
▼

Assign small memories explicitly in the initial block. Veloce creates a corresponding contents file automatically.

Explicit + loop initialization
VERILOG
bit [31:0] mem2 [7:0]; // pragma attribute mem2 ram_block 1 bit [31:0] mem3 [7:0]; // pragma attribute mem3 ram_block 1 initial begin for (int i = 0; i < 8; ++i) mem2[i] = {32*i+1'b1}; // Remaining mem3 locations initialized with 0's mem3[1] = 32'd7; mem3[5] = 32'd9; mem3[7] = 32'd11; end
🧬 SystemVerilog Array Literal Initialization
▼

Use SystemVerilog array literals to specify initial contents. Veloce creates a corresponding contents file and adds an entry to the vmw_mem_contents file.

SV array literal
SYSTEMVERILOG
// Initialize all 8 locations with value 37 bit [31:0] mem4 [7:0] = {8{32'd37}}; // pragma attribute mem4 ram_block 1
📁 vmw_mem_contents File Format
▼

You can override memory contents at emulation time by writing a new vmw_mem_contents file:

vmw_mem_contents format
PLAINTEXT
# Initialize all instances of a descriptor DESCRIPTOR descriptor_name contents_file content_format # Initialize a specific instance only INSTANCE instance_name contents_file contents_format

Memories in Combinational Cycles
Velsyn does not support memories found as part of a combinational cycle and will exit compilation with an error.
🚫 Only memories with ordered write semantics and level sensitivity can form part of combinational cycles. When velsyn finds a memory in a combinational cycle, it issues an error and exits compilation.

Example Error Messages

Velsyn combinational cycle errors
ERROR
Error [Velsyn-14040]: Path top.wram: Memory appears to be in a combinational cycle via terminal WDO[1]. Error [Velsyn-14040]: Path top.sram: Memory appears to be in a combinational cycle via terminal RDO[0]. Error [Velsyn-14040]: Path top.sram: Memory appears to be in a combinational cycle via terminal WDO[1].
⚠️ Limitation 1
Combinational cycles carried in by the address terminals of memories are not reported. However, associated read port data terminals that carry the cycle out are reported.
⚠️ Limitation 2
Combinational cycles carried in by byte enable terminals are instead reported as cycles carried through the write enable terminals.
✅ Workaround Pragma
Use ignore_read_write_priority pragma on the register declaration to make Veloce infer a level unordered memory and avoid false combinational loop errors.
ignore_read_write_priority pragma
VERILOG
// pragma attribute <registerName> ignore_read_write_priority 1 reg [31:0] core [63:0]; // 64x32 bit memory array // pragma attribute core ram_block 1 // pragma attribute core ignore_read_write_priority 1
⬆️
Uploading Multiple Memories in Parallel
Veloce supports parallel memory upload to improve performance when working with multiple memories simultaneously.
💡 Parallel upload allows you to upload multiple memories at the same simulation time, significantly improving throughput for designs with many large memories.
⚡ Deferred Upload (Default)
By default, Veloce defers memory upload — contents are not uploaded immediately but batched for better performance. Only modified pages are written to save disk space.
🔄 Disable Deferred Upload
To force immediate upload always, set the deferred upload flag to "true" in the Veloce config. Useful for debugging memory state at precise simulation times.
📦 Full Memory Dump
By default only modified pages are uploaded. Use -flexmem_dump_full_memory at runtime to force a complete memory dump including unmodified pages.
Force full memory upload at runtime
BASH
# Upload all pages including unmodified ones velrrun -c -flexmem_dump_full_memory -do run.do

Out-of-Bounds Access on Signals and Memories
Veloce defines specific behavior for out-of-bounds reads and writes on signals and memories.
✏️ Out-of-Bounds Writes
Out-of-bounds writes are silently ignored. No error is generated at runtime. The memory contents remain unchanged.
Silently Ignored
📖 Out-of-Bounds Reads
Out-of-bounds reads return 'X' in simulation but return 0 in Veloce emulation. This applies regardless of whether the signal is mapped to gates or memories.
Returns 0 in EmulationX in Simulation
🔍 Compile-Time Detection
By default, no compile-time information is provided for out-of-bounds accesses. Use a compile switch to enable a SimError during the rtic phase for out-of-bounds read detection.
Optional Flag
ℹ️ Out-of-bounds behavior is independent of where the signal is implemented — it applies equally to signals mapped to gates, standard memories, or Flexmem memories.

Flexmem
Flexmem partitions large memory images between the HDL emulator and HVL host — keeping only a small cache on hardware while the full image resides on the HVL side.

🧠 Problem 1: Capacity
Large system-level memories mapped entirely into emulator hardware cause significant capacity impact, long compile times, and sub-optimal clock speeds.
🔗 Problem 2: HVL Access
Frequent HVL-side access to memories inside the DUT causes significant communication overhead that impacts runtime throughput.
✅ Flexmem Solution
Automatically partitions the memory image. Full image on HVL side for fast access. HDL holds only a small cache — optimizing both compile time and runtime.

Specifying Flexmem

XRTl pragma — Flexmem declaration
XRTL
// pragma attribute XMEM flexmem cache [options] bit [7:0] XMEM [1<<36];
Verilog attribute — Flexmem declaration
VERILOG
(* flexmem="cache", [options] *) int RAM [1 << 25];
💡 Providing an empty string "" or "null" as the Flexmem type makes the memory non-Flexmem. This lets you keep smaller memories as non-Flexmem while larger ones use Flexmem.
Parameterized Flexmem values
VERILOG
parameter P_PC = 100; (* flexmem="cache", page_count=P_PC, page_size_in_bytes=2048 *) int RAM [1 << 25];

Comodel Transaction-Based Upload/Download for SBMEMs
Specify at compile time that SBMEM upload and download should use comodel transactions for improved performance.
⬇️ Download Performance
Memory download using comodel transactions is faster when the number of locations to be downloaded is small. Avoids full-memory transfer overhead.
⬆️ Upload Performance
Memory upload using comodel transactions improves performance by allowing you to upload multiple memories at the same simulation time.
🔀 Multi-Comodel Mapping
In multi-comodel mode, Flexmem HVL images and communication traffic can be distributed across multiple secondary comodel processes for better bandwidth.
⚠️ Once a comodel process exits (e.g., due to $finish or C-side exit()), you cannot use Tcl memory commands on Flexmems. The comodel process keeps the complete Flexmem HVL memory image. Use a Verilog final block for memory upload in such cases.

Ternary Content-Addressable Memory (TCAM) Modeling
TCAM extends CAM by allowing "don't care" matching using two-bit representation per data bit — modeled as two memory signals in RTL.
🔍 CAM
Content-Addressable Memory searches for data content in memory and returns the memory address(es) containing that data. Reverse of a normal memory lookup.
🔀 TCAM
Extends CAM with "don't care" matching. Uses two-bit representation per data bit — modeled as two memory signals (MEMORY_A and MEMORY_B) in RTL.
📊 Encoding Scheme
DB=0, DA=0 → X (don't care) · DB=1, DA=0 → 0 · DB=0, DA=1 → 1 · DB=1, DA=1 → Invalid. Two bits encode four states.

TCAM Encoding Table

DB (MEMORY_B)DA (MEMORY_A)Encoded ValueMeaning
00XDon't care — matches both 0 and 1
100Matches only 0
011Matches only 1
11InvalidNot a valid encoding

TCAM Write Operations

TCAM Write — Basic RTL
VERILOG
reg [DATA_WIDTH-1:0] MEMORY_A [DEPTH_WORDS-1:0]; reg [DATA_WIDTH-1:0] MEMORY_B [DEPTH_WORDS-1:0]; always@(posedge CLK) begin if (WE) begin MEMORY_B[WRADDR] = DATA_B; MEMORY_A[WRADDR] = DATA_A; // valid bit population to track valid addresses VBIT[WRADDR] = 1'b1; end end
TCAM Read — Basic RTL
VERILOG
// read operation on TCAM always@(*) begin RD_DATA = (Rd_En) ? MEMORY_B[RDADDR] : MEMORY_A[RDADDR]; end
🚀
TCAM Modeling for Veloce
Optimized Veloce TCAM implementation using $cam_write and $cam_read system calls, plus the CAMUpload C API.
💡 Replace dual-memory RTL with a single CAM_MEMORY array marked with tcam_block 1 pragma. Use $cam_write and $cam_read for optimized emulation performance.

Optimized Write with $cam_write

Veloce-optimized TCAM write
VERILOG
reg [(2*DATA_WIDTH)-1:0] CAM_MEMORY [DEPTH_WORDS-1:0]; //pragma attribute CAM_MEMORY tcam_block 1 always@(posedge CLK) begin if(WE) begin $cam_write(CAM_MEMORY, WRADDR, DATA_B, DATA_A); VBIT[WRADDR] = 1'b1; end end

Optimized Read with $cam_read

Veloce-optimized TCAM read
VERILOG
reg [(2*DATA_WIDTH)-1:0] CAM_MEMORY [DEPTH_WORDS-1:0]; //pragma attribute CAM_MEMORY tcam_block 1 reg[DATA_WIDTH-1:0] Temp_RdDataA, Temp_RdDataB; always@(*) begin $cam_read(CAM_MEMORY, RDADDR, Temp_RdDataB, Temp_RdDataA); RD_DATA = (Rd_En) ? Temp_RdDataB : Temp_RdDataA; end

CAMUpload C API

TbxManager::CAMUpload signature
C++
bool TbxManager::CAMUpload( const char* instName, // TCAM hierarchical path in RTL const char* maskFile, // mandatory mask file name const char* dataFile, // mandatory data file name int start_addr = -1, // default: LSB of TCAM depth int end_addr = -1, // default: MSB of TCAM depth CAMDataFormat eFormat = DF_HEX // DF_BIN or DF_HEX (default) );
ℹ️ Supported data formats for CAMUpload are binary (DF_BIN) and hex (DF_HEX) only. Default format is hex. Runtime Tcl commands cam upload and cam download are also supported.

TCAM Auto-Inference Cases

1
Two Separate Memories (MEMORY_A + MEMORY_B)
Memory is inferred automatically. No extra marking pragma required. Veloce detects the dual-memory TCAM pattern.
2
Single Memory — Side by Side or Vertical
A single memory used for both Mask and Data placed side by side or vertically. No extra marking pragma required.
3
Single Memory — Consecutive Addresses
A single memory where Mask and Data are placed on consecutive addresses. No extra marking pragma required.
4
Issues? Use tcam_block Pragma
If auto-inference fails, manually mark the TCAM memory signal with // pragma attribute CAM_MEMORY tcam_block 1 or use the remodeling methodology.
