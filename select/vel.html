Chapter 1 — Veloce Emulation Overview

Hardware-assisted verification systems for SoC and embedded system designs.

Key Concepts
📌 Key Concept What is Veloce?
Veloce emulators are hardware-assisted verification systems for SoC and embedded system designs. They provide:
  • Transaction-based simulation acceleration
  • In-Circuit Emulation (ICE)
  • VirtuaLab emulation
  • Targetless emulation
Used to accelerate concurrent hardware/software validation and embedded software validation.
📌 Key Concept Hardware Environments
Veloce emulators operate in two environments:
  • Local hardware environment — compile, runtime, and comodel hosts on local machines
  • Data center environment — compile server, distributed compute, storage servers on a network (accessible globally via NFS)
Key platforms: Veloce StratoM, Strato T, StratoCS, StratoPlus, PrimoCS
📌 Key Concept Use Modes
  • Transaction-Based Acceleration — SCE-MI standard; DPI import/export, transaction pipes, TLM FIFOs
  • ICE (In-Circuit Emulation) — IO ports/cables for real-world stimulus; iSolve™ Solutions
  • Target-less — no physical target device required
  • VirtuaLAB — transistor-level virtual peripheral models; no physical connection needed
  • Software Development — Sourcery CodeBench, Veloce Virtual Probe, Codelink, Warpcore
⚠️ All use modes require a comodel host.
🔄 Workflow General Emulation Flow
High-level flow (in order):
  1. Create libraries → velllib
  2. Map libraries → velmap
  3. Create veloce.config file
  4. Analyze RTL files → velanalyze
  5. Compile DUT → velcomp
  6. Compile testbench → velhvl or vlog
  7. Run emulation → velrun or vsim
  8. Debug → velview / Visualizer

Chapter 2 — Getting Started

Setting up your environment and design for emulation.

Environment Setup
⚙️ Config Required Environment Variables
Add these to your setup.csh resource file: 
setenv VELOCE_HOME <install_dir>/Veloce_vx.x.x
setenv MED_EMULATOR <path_to_EMULATORS_dir>
setenv LM_LICENSE_FILE <path_to_license_server>
# Add to PATH:
setenv PATH $VELOCE_HOME/bin:$PATH
⚠️ Warning OS Version Dependency
Veloce runtime has a hard dependency on the exact OS version due to the PHB card driver. Only the exact 2-digit OS version is explicitly tested. Subsequent minor versions are expected to be compatible but not guaranteed.
⚠️ Warning TCP/IP Port Requirement
Ephemeral TCP/IP ports (32768–61000) must be open or Veloce will error when running emulation with either velrun or vsim.
⚠️ Warning NFS Requirement for Compile Database
The compile database (or any file required for emulation) cannot be kept on local disk of any specific machine. Veloce requires NFS access to the database from comodel hosts and emulator hosts.
Design Setup
🔄 Workflow Setting Up Your Design
  1. Create a design directory
  2. Prepare HDL and HVL source files
  3. Set up and map analysis libraries (velllib, velmap)
  4. Create a veloce.config file with all required parameters
# Using a libmap file for analysis:
velanalyze -f <command_file> -libmap lib.map
# Example:
velanalyze -f file.f
⚙️ Config veloce.config — Phase Options
Config file phase names and their purpose:
  • comp — compilation options (clocks, assertions, RTLC, C compile, job distribution)
  • veanalyze / vhanalyze — RTL source file analysis options
  • rtlc — RTL compiler options
  • velsyn — synthesis utility options
  • velcc — crystal chip timing database generation
  • velgs — global clock scheduler options
  • hvl — DPI-C file compile options
  • source <file> — include another config file
⚙️ Config veloce.config — Full Example 
comp -platform StratoCS
comp -num_boards 5
comp -top sys
comp -top_hdl verilog

rtlc -partition_module_xrtl sys
rtlc -partition_module_xrtl iotool
rtlc -opt_best

comp -distrib_flow grid

hvl -cflags "-I./sim/include"
hvl -ldflags "-L./sim/tbx -I./linux -lusertb"
comp -cui_file vmw.cui
⚠️ Warning Config File — Critical Rule
You must specify ALL required parameters for compilation in the veloce.config file before compilation. You cannot modify or specify parameters during compilation.

⚠️ Using -svf or -vef with -libmap results in an error.

Chapter 3 — Compiling Your Design

Compile-time features, options, and optimization techniques.

Core Compile Commands
⚡ Command velanalyze — Analyze RTL Source Files
Analyzes RTL source files before or during compilation. 
# Analyze SystemVerilog files:
velanalyze hdl/*.sv

# Analyze with a command file:
velanalyze -f file.f

# Analyze with libmap:
velanalyze -f file.f -libmap lib.map

# One-step (in veloce.config):
veanalyze <files>    # Verilog
vhanalyze <files>   # VHDL
⚡ Command velcomp — Compile DUT
Main compilation command. Run from top level of design directory. 
# Basic compile:
velcomp

# Compile with top module specified:
velcomp -top <top_module>

# Single-step (analyze + compile):
velcomp -top <top_module>   # with comp -single_step in config

# Start at specific task:
velcomp -start_task <task_name>

# Run only velvis task:
velcomp -task velvis

# Lock/unlock project (before export):
velcomp -lock_project
velcomp -unlock_project

# Export compile database:
velcomp -create_export_db /path/to/exportedDir

# Clean and restart a task:
velcomp -clean_task velcc
velcomp -start_task velcc
⚡ Command velhvl — Compile HVL Testbench
Compiles C/C++/SystemC HVL models into a comodel executable. 
# Compile C testbench:
velhvl -cfiles hvl/hello.cxx

# With include flags:
velhvl -cfiles mytest.cxx -cflags "-I./include"

# With link flags:
velhvl -cfiles mytest.cxx -ldflags "-L./lib -lmylib"
💡 velhvl reads veloce.config and uses options under the hvl phase. Command-line options take precedence over config file.
🔄 Workflow Two-Step vs One-Step Compile
Option A — Two-Step (separate analyze + compile): 
velanalyze hdl/*.sv    # Step 1: analyze
velcomp               # Step 2: compile (starts from RTL phase)
Option B — One-Step (analyze + compile together): 
# In veloce.config add:
comp -single_step
veanalyze <files>

# Then run:
velcomp -top <top_module>
⚙️ Config Exporting a Compiled Database 
velcomp -lock_project
velcomp -create_export_db /path/to/exportedDir
velcomp -unlock_project
Allowed on exported DB: HVL compiles, shadow creation, emulation run, emulation debug.
NOT allowed: velanalyze or velcomp.
💡 Tip Improving Compile Time
  • Parallel velsyn — run velsyn in parallel threads
  • Distributed velsyn — distribute across farm machines
  • Template Processing — process repeated module hierarchy once; reduces memory and compile time for large designs (2M/3M/4M gate Strato)
  • RTLC Templates — reuse RTLC results across identical instances
Add to veloce.config: 
comp -distrib_flow grid    # or lsf / custom
comp -single_step
⚠️ Warning velcc Partitioning/Routing Issues
Before generating a CUI file for routing issues, check:
  • Did you use rtlc advisor capOnly/perfOnly/capAndPerf/aggressive? These make partitioning harder.
  • Did you use -Mm higher than 10.0? Not recommended — causes routing difficulty.
  • Did you use velsyn -enableSoftCMEMF? Consider removing it — increases routing congestion.
⚙️ Config Enable Critical Path Analysis (velcp) 
# In veloce.config:
comp -enable_velcp

# If already compiled without it:
velcomp -clean_task velcc
velcomp -start_task velcc
velcp
⚠️ Increases compile database size by ~15%. Negligible effect on compile time/memory.

Chapter 4 — Running Emulation

Runtime use modes, commands, and examples.

Runtime Commands
⚡ Command velrun — Runtime Use Modes 
# Auto-run (connect, download, run, exit):
velrun

# Interactive Tcl shell:
velrun -c

# Interactive Tcl shell (no auto-connect):
velrun -c -nac

# GUI mode (Tcl shell + debug window):
velrun -gui

# Run from do file:
velrun -c -nac -do cmd.do

# Questa testbench:
vsim hdlTop hvlTop TbxSvManager
⚡ Command velrun — Special Use Modes 
# Download only (manual connect):
velrun -download

# Disconnect a downloaded session:
velrun -disconnect

# Pass args to comodel executable:
velrun -exe_args "-enable_display"

# Specify custom comodel executable:
velrun -exe hello_world -exe_args "-myOpt1 -myOpt2"
⚡ Command Runtime Examples — Full Sequences 
# Basic run sequence:
velrun -c
% hwtrace autoupload on -tracedir abc
% run 1000000 ns
% upload wave ..
% quit

# With memory and trigger:
velrun -c
% memory download ..
% trigger download vmw.trig
% run
% waitfor runcomplete
% upload wave ..
% quit

# Multiple tests:
velrun -gui -exe_args "-mytest1"
run
setvar comdel_exe_args "-mytest2"
reinitialize
run
setvar comdel_exe_args "-mytest3"
reinitialize
quit
⚡ Command Questa Runtime Commands 
vsim hdlTop hvlTop TbxSvManager
% velocecmd "trigger download file"
% run X ns
% velocecmd "upload -tracedir abc"
% quit

# Or with velrungui:
vsim -c
% log -r *
% run 100 ns
% velocecmd "reg getvalue signame"
% run 100 ns
velrungui open
% hwtrace autoupload on -tracedir abc
% reg setvalue signame
% display off
% velrungui close
% run 200 ns

P0 => AVB3_0
P1 => AVB3_1

# Alter with: comp -comodelLayout
💡 This keeps config independent of hard-coded connector names.

Chapter 5 — Debugging

Waveform capture, upload, triggers, and runtime behavior debugging.

Waveform Capture
📌 Key Concept Waveform Capture Methods
Three methods to capture waveforms:
  • Time-based — know the timepoint where design mismatches
  • Line Breakpoint (LBP) — know the RTL source line; StratoCS does NOT support this
  • Trigger-based — capture based on signal state/event
⚠️ Emulator only captures waveforms when Trace is ON. Default: Trace is ON after design download.
⚡ Command Trace ON / OFF 
# From Tcl shell:
hwtrace on
hwtrace off

# Auto-upload with trace directory:
hwtrace autoupload on -tracedir <dir>

# Recover interrupted conversion:
hwtrace autoupload on -recover
Via GUI: DesignFlow > Emulation > Set Trace On
🔄 Workflow Time-Based Waveform Capture
  1. Invoke GUI: velrun -gui
  2. Ensure Trace is ON: DesignFlow > Emulation > Set Trace On
  3. Run to timepoint of interest: run 10000 ns
  4. Upload waveform (see Upload Waveforms)
  5. View in Wave Viewer: DesignFlow > Debug > Wave Viewer
🔄 Workflow Viewing Waveforms
  1. Open Wave Viewer: DesignFlow > Debug > Wave Viewer
  2. If after runtime: File > Specify Dataset → navigate to Trace directory
  3. Load signal list: File > Load Signal List or drag from Objects tab
  4. View waveforms in Wave Viewer
# View RTL triggers with DUT:
vis -veloc -veltrigger <trigger_db_name>

# Load DUT + all RTL trigger hierarchies:
vis -tracedir <trace_directory_name>
⚡ Command Convert to CRD / FSDB Format 
# Convert trace to CRD (shareable, no reconstruction needed):
ecf2wave -tracedir <dir> -crd -o output.crd -siglist <sigfile>

# Generate FSDB offline:
ecf2wave -tracedir <dir> -fsdb
💡 CRD datasets can be exported as tarred zip for remote debugging teams.
📌 Key Concept Waveform Special Values
  • X — unknown value
  • Z — high impedance
  • NA — not available (trace buffer not filled for that chip); propagates as X
⚠️ Low power X does not further propagate. Propagation from flops to combinational logic is based on real emulation values.
📌 Key Concept Backup-Replay & Testbench Restore
Two methods for deterministic debugging:
  • Backup-Replay — captures software/testbench input transactions → replay later with design on emulator
  • Replay-Based Testbench Restore — captures emulator output transactions → replay later with software on comodel machine
Unified interface: xcapture and xreplay commands.

LSF, Grid Engine, and custom job distribution setup.

Distribution Setup
⚙️ Config Enable Grid Distribution
Add to veloce.config: 
comp -distrib_flow grid
# Specify cluster queue:
# (default: -q all.q)

comp -distrib_flow custom

Chapter 9 — Triggers

Control hardware trace buffers during runtime to capture waveform data efficiently.

Trigger Types & Usage
📌 Key Concept Trigger Types
Flexible RTL Triggers (newer, recommended):
  • Specified using standard SystemVerilog (RTL subset)
  • Signal list provided in trigger input file during main design compile
  • Quick trigger compile without full re-compile (StratoPlus only)
  • Runtime Tcl commands for control/observation
Runtime Triggers (older):
  • Proprietary HDL-like language
  • Limited expressions; only ONE active trigger state machine
  • Trigger input file provided at runtime
⚡ Command Trigger Examples 
// Store trigger (captures waveforms, doesn't stop design):
domain domain0 @ (posedge Uclock);
state S0 : initial, store
begin
end
next (top.SIM_TIME[63:0] == 100);
state S1 : trigger, store;

// Stop trigger (stops design when trigger matures):
state S0 : initial, store
begin
next (top.SIM_TIME[63:0] == 100);
end
state S1 : trigger, stop;
🔄 Workflow Trigger-Based Waveform Capture
  1. Create a trigger file
  2. Invoke GUI: velrun -gui
  3. Ensure Trace is ON: DesignFlow > Emulation > Set Trace On
  4. Download trigger: DesignFlow > Emulation > Download Trigger
  5. Run: run
  6. Upload and view waveform
# Load RTL trigger at runtime:
velrun ... -veltrigger sync_trig_part_1
📌 Key Concept Trigger Position
Trigger position controls where in the captured waveform the trigger event falls:
  • 50 (default) — trigger at midpoint; half before, half after
  • 0 — trigger at start; window mostly after trigger
  • 100 — trigger at end; window mostly before trigger
⚠️ After a trigger matures, Trace is automatically turned OFF. You must set Trace ON again for next capture.

Chapter 10 — Memory Modeling

Parameterized memories, Flexmem, TCAM, and memory optimization.

Memory Types & Concepts
📌 Key Concept Parameterized Memory Interface
Veloce recognizes a module as memory when it has VMW_PARAM_MEM prefix and exactly 6 parameters in order:
  1. address width
  2. data width
  3. write semantics
  4. output enable
  5. number of read ports
  6. number of write ports
⚠️ Any number other than 6 with the VMW_PARAM_MEM prefix → velsyn reports an error.
📌 Key Concept Flexmem Overview
Flexmem partitions memory between HDL and HVL sides:
  • HVL side — full memory image (sparse, page-based)
  • HDL side — small cached copy (<1% of total memory)
Benefits: Eliminates capacity issues for large memories; fast HVL access; transparent data sync; supports $readmem/$writemem.
hd page size: configurable, typically 128B–16KB.
⚠️ Warning Flexmem — Important Limitations
  • Do NOT use 0-time export DPIs with Flexmem → use back-door access with SCEMI-DMIs instead
  • Small memories (<=16MB) with primarily HDL access are NOT good candidates for Flexmem
  • Do NOT use -enableSoftCMEMF option with Flexmem → significant drop in compile frequency
  • Reads/writes inside synchronous always blocks with asynchronous set/reset are NOT supported
  • Asynchronous writes from combinational always blocks are NOT supported
💡 Tip Flexmem — Optimal Channel Size
Keep transaction word size a multiple of memory word size, or use 32 or 64 (default: 64 words of 32-bit each).

Example: 128-bit memory word → use transaction size of 32 (32 × 32-bit = 1024-bit = 128 bytes = 8 memory words). This maximizes comodel channel throughput.
⚡ Command TCAM Modeling — Optimized Implementation 
// Pragma to mark TCAM memory:
//pragma attribute CAM_MEMORY tcam_block 1

// Optimized TCAM write:
$cam_write(CAM_MEMORY, WRADDR, DATA_B, DATA_A);

// Optimized TCAM read:
$cam_read(CAM_MEMORY, RDADDR, Temp_RdDataB, Temp_RdDataA);
⚠️ Requires a Veloce OS license depending on platform.
📌 Key Concept Out-of-Bounds Access Behavior
  • Out-of-bounds writes → ignored
  • Out-of-bounds reads → 'X' in simulator, 0 in Veloce emulation
To enable compile-time error for out-of-bounds reads, add to config: 
rtlc -oob_read_error

Chapter 11 — Performance & Capacity Optimization

General optimization, HDL optimization, concurrent modeling, and hardware configuration.

RTLC Advisor & HDL Optimization
📌 Key Concept RTLC Advisor
Automates cross-module capacity and performance optimization. Modes:
  • capAndPerf — optimize for both capacity AND performance (recommended, 2-step)
  • capOnly — capacity only
  • perfOnly — performance only
Output: Performance analysis report + critical capacity modules list in compile directory.
⚡ Command RTLC Optimization Commands 
# Flattening options:
rtlc -flatten_mod <module>
rtlc -flatten_hier <module>
rtlc -flatten_netlist_mod <module>

# Retiming:
rtlc -ena_retime_shift_delay
rtlc -retime_annot
comp -rtlc_advisor_options "-retime_opt"

# Interleaved AMCP partitioning:
rtlc -interleaved_amcp

# One-way import DPI optimization:
rtlc -one_way_caller_opt          # global
rtlc -one_way_caller_opt_mod <mod>  # per module
rtlc -no_one_way_caller_opt_mod <mod>  # disable per module
🔄 Workflow RTLC Optimization Re-Use Flow
Save compile time in subsequent compilations:
  1. Disable RTLC Advisor in veloce.config
  2. Copy the feedback directory to a user area (velcomp deletes it at start)
  3. Point to copied directory with: rtlc -feedback_dir <path>
  4. Run velcomp
  5. After each compile, copy the new feedback directory — quality improves each iteration
💡 Tip Guidelines for Better Performance
  • Use $display tasks sparingly — they impact performance
  • Use Inbound Streaming DPIs for streaming data from files/sockets (avoids reactive DPI overhead)
  • Use AOWC optimization (enabled by default) for one-way hardware-to-software communication — can provide up to 2X performance gain
  • Use Concurrent HDL/HVL modeling for up to 2X+ gain over reactive DPI systems
  • For Flexmem: avoid 0-time export DPIs; use SCEMI-DMIs for back-door access
⚙️ Config Frequency Configuration
Change runtime frequency in veloce.config: 
comp -freq <value>
# Supported on Strato and StratoCS platforms
Precision vs Minimum Clock Frequency:
  • 1ps precision → min 62 MHz
  • 10ps → min 6.2 MHz
  • 100ps → min 620 KHz
  • 1ns → min 62 KHz
⚠️ Warning RTLC Advisor — Important Restrictions
  • Do NOT perform design analysis in subsequent runs after advisor
  • Do NOT change RTLC elaboration options between runs
  • Do NOT change velsyn control files and options between runs
  • Different compile flows (power, DFT, gate-level, PCR) are treated as separate entries — recommendations do NOT cross flows
  • Aggressive capacity optimization optimizes state elements → those signals become inaccessible at runtime unless preserved
